\section{Grundlegende Problemstellung}
% =============================================================================
\vspace{.25cm}
Bei der Ausübung von Krafttraining ist die korrekte Bewegungsausführung entscheidend für den Trainingserfolg und die Verletzungsprävention. Trainierende erhalten jedoch selten \newline unmittelbares, objektives Feedback zu ihrer Bewegungsqualität. \vspace{.25cm}

\textbf{MoveLink} adressiert dieses Problem: Trainingsgeräte (z.\,B.\ Hanteln) werden mit Sensoren \newline ausgestattet, die Bewegungsdaten in Echtzeit erfassen und drahtlos an eine Anwendung \newline übertragen. Dort wird dem Trainierenden visuelles Feedback gegeben. \vspace{.25cm}

Im Rahmen dieses Laborprojekts wird ein \textbf{Proof of Concept} entwickelt, der die gesamte \newline Datenkette -- von der Sensorerfassung über die Übertragung bis zur Echtzeit-Visualisierung -- prototypisch umsetzt. Die Bewegungserkennung via Machine Learning ist Gegenstand eines separaten Projekts und wird hier als Datenquelle abstrahiert.
\vspace{.25cm}

\section{Architekturvorschlag}
% =============================================================================
\vspace{.25cm}
\subsection{Gewählte Architektur: Client-Server mit BLE-Gateway}
\vspace{.25cm}
Die Architektur folgt einem \textbf{dreischichtigen Client-Server-Modell}:
\vspace{.5cm}
\begin{figure}[H]
    \centering
    \begin{tikzpicture}[
        box/.style={draw, rounded corners, minimum width=3cm, minimum height=1cm, align=center, font=\small},
        arrow/.style={->, thick, >=stealth},
        label/.style={font=\scriptsize, midway, above}
    ]
        \node[box, fill=hkared!15] (sensor) at (0, 0) {XIAO nRF52840\\+ IMU-Sensoren};
        \node[box, fill=hkared!25] (server) at (5.5, 0) {Backend-Server\\(Gateway / API)};
        \node[box, fill=hkared!35] (frontend) at (11, 0) {Frontend\\(Web-App)};
        \draw[arrow] (sensor) -- node[label] {BLE} (server);
        \draw[arrow] (server) -- node[label] {WebSocket} (frontend);
        \draw[arrow] (frontend) -- node[label, below] {REST} (server);
    \end{tikzpicture}
    \caption{Übersicht der Systemarchitektur}
\end{figure}
\vspace{.25cm}
\begin{itemize}[leftmargin=*, nosep]
    \item \textbf{Embedded:} XIAO nRF52840 erfasst Bewegungsdaten (Accelerometer, Gyroskop) \newline und sendet sie via \textbf{BLE}. \\
    \item \textbf{Backend:} Lokal in einem Docker-Container betriebener Server als BLE-Gateway; \newline leitet Daten über \textbf{WebSockets} weiter und bietet eine \textbf{REST-API} für Konfiguration. \\
    \item \textbf{Frontend:} Web-Applikation zur Echtzeit-Visualisierung der Bewegungsdaten.
\end{itemize}
\vspace{.5cm}
\subsection{Diskussion von Alternativen}

\begin{table}[H]
\centering
\small
\begin{tabularx}{\textwidth}{lXX}
    \toprule
    \textbf{Alternative} & \textbf{Vorteile} & \textbf{Nachteile} \\
    \midrule
    \textbf{Direkt BLE $\rightarrow$ App} (kein Server)
        & Geringere Latenz; \newline weniger Infrastruktur
        & Web Bluetooth API \newline eingeschränkt; \newline plattformabhängig; \newline keine zentrale Verarbeitung \\
    \midrule
    \textbf{MQTT-Broker} (IoT-Standard)
        & Skalierbar; \newline Pub/Sub ideal für \newline Sensordaten
        & Infrastruktur; \newline Overhead für PoC \newline überproportional \\
    \midrule
    \textbf{Native App} (Flutter/MAUI)
        & Direkter BLE-Zugriff; \newline native Performance
        & Aufwand; \newline schwerer im Team zu \newline parallelisieren \\
    \bottomrule
\end{tabularx}
\caption{Vergleich der Architektur-Alternativen}
\end{table}
\vspace*{-.75cm}
\textbf{Begründung:} Die Client-Server-Architektur ermöglicht eine klare Trennung der Verantwortlichkeiten (Embedded\,/\,Backend\,/\,Frontend), erleichtert parallele Teamarbeit und liefert eine plattformunabhängige Web-Oberfläche. Das gesamte System wird \textbf{lokal} über Docker Compose betrieben -- es ist kein externer Server oder Cloud-Hosting erforderlich.

\section{Use Cases}
% =============================================================================
\vspace{.25cm}
\textbf{UC-1: Echtzeit-Training überwachen} (Primär) \vspace{.25cm} \newline 
\textit{Akteur:} Trainierender \quad \newline\textit{Vorbedingung:} Sensor verbunden \vspace{-.25cm} \newline
\begin{enumerate}[nosep]
    \item Trainierender startet eine Trainingseinheit in der App. \\
    \item Sensor erfasst Bewegungsdaten und sendet sie via BLE an den Server. \\
    \item Server leitet die Daten per WebSocket an das Frontend. \\
    \item Frontend visualisiert die Bewegung in Echtzeit ($<$ 1\,s Latenz).
\end{enumerate}
\vspace{.5cm}
\textbf{UC-2: Sensorverbindung herstellen}\\ \vspace{-.25cm} \newline
\textit{Akteur:} Trainierender \quad \newline\textit{Vorbedingung:} Sensor eingeschaltet, Server läuft \vspace{-.25cm} \newline
\begin{enumerate}[nosep]
    \item App zeigt verfügbare Sensoren an. \\
    \item Trainierender wählt Sensor aus; Verbindung wird hergestellt.
\end{enumerate}
\vspace{.5cm}
\textbf{UC-3: Trainingsdaten einsehen} (Sekundär)\\ \vspace{-.25cm} \newline
\textit{Akteur:} Trainierender \quad \newline\textit{Vorbedingung:} Vergangene Trainingseinheit vorhanden \vspace{-.25cm} \newline
\begin{enumerate}[nosep]
    \item Trainierender navigiert zur Übersicht vergangener Einheiten. \\
    \item App zeigt historische Bewegungsdaten an.
\end{enumerate}

\section{Muss- und Kann-Kriterien}
% =============================================================================
\vspace{1.5cm}
\textbf{Muss-Kriterien:} \vspace{1cm}
\begin{enumerate}[label=\textbf{M\arabic*}, nosep]
    \item Echtzeitübertragung der Sensordaten mit einer Latenz von \textbf{max.\ 1\,s}. \\
    \item \textbf{Visuelle Darstellung} der Bewegungsdaten als Live-Diagramm. \\
    \item \textbf{Eigenständige Web-Applikation} (kein Plugin / nativer Client nötig). \\
    \item Plattformübergreifend nutzbar: \textbf{Android (Browser), macOS, Windows}. \\
    \item Klare \textbf{Trennung der Systemkomponenten} (Embedded, Backend, Frontend). \\
    \item \textbf{Lokaler Betrieb} des gesamten Systems via Docker Compose (kein Cloud-Hosting).
\end{enumerate}

\vspace{1.5cm}
\textbf{Kann-Kriterien:} \vspace{1cm}
\begin{enumerate}[label=\textbf{K\arabic*}, nosep]
    \item Einfache \textbf{Bewertung der Bewegungsausführung} (Schwellenwert-basiert). \\
    \item \textbf{Persistierung} von Trainingsdaten für spätere Einsicht. \\
    \item \textbf{3D-Visualisierung} der Sensorbewegung (z.\,B.\ Three.js). \\
    \item Unterstützung \textbf{mehrerer Sensoren} gleichzeitig. \\
    \item \textbf{Konfigurierbarkeit} der Darstellung (Achsen, Zeitfenster).
\end{enumerate}
